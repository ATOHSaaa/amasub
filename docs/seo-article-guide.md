## 2. SEOの基本方針

### 検索意図を先に決める

記事を書く前に、次を1文ずつ書き留める（内部メモで可）。

- **メインキーワード**（例: `三体 Kindle Unlimited`、`Audible 変な家`）
- **検索意図**（Know / Do / Compare のどれか）
- **読者が記事読了後に得る答え**（例: 現時点で読み放題か、確認方法、代替案）

### キーワードの置き方

- **タイトル（`title`）**: メインキーワードを前半に。32〜40文字目安（長すぎると検索結果で切れる）
- **メタ説明（`description`）**: キーワードを自然に含め、120〜160文字程度。クリックしたくなる要約
- **見出し**: H2に関連語を含める。H3で細分化
- **本文初段**: メインキーワードと結論の要点を80〜150字で述べる（結論ファースト）
- **本文**: 同語の詰め込みは禁止。関連語・共起語で網羅性を出す
- **URL（ファイル名）**: 英小文字・ハイフン。例: `santai-kindle-unlimited.md`

### タイトル公式（テンプレ）

| タイプ | 例 |
|--------|-----|
| 作品×Kindle Unlimited | `『作品名』はKindle Unlimitedで読み放題？配信状況・確認方法を解説` |
| 作品×Audible | `『作品名』はAudibleで聴き放題？配信状況・聴くメリットを解説` |
| 比較 | `AudibleとKindle Unlimitedの違いを比較｜どっちを選ぶべき？` |
| 活用法 | `Kindle Unlimitedで〇〇を効率よく楽しむ方法｜おすすめの使い方` |

区切りは `｜`（全角）を推奨。`【】` の乱用は避ける。

## 3. フロントマター（必須）

記事は `src/content/posts/` に Markdown で配置する。

```yaml
---
title: '検索結果に表示されるタイトル'
description: 'メタディスクリプション。120〜160文字目安。'
pubDate: 2026-06-01
updatedDate: 2026-06-15  # 内容を大きく更新したときのみ
author: アマサブ編集部
services:
  - kindle-unlimited   # または audible。複数可
tags:
  - 作品名やテーマ     # 関連記事マッチ用
cta: kindle-unlimited # 省略可。both | none も可
draft: false
---
```

| フィールド | ルール |
|-----------|--------|
| `services` | 記事の主題サービスを必ず指定。比較記事は両方 |
| `tags` | 作品名・著者名・ジャンルなど2〜5個 |
| `cta` | 未指定時は `services` から自動。比較記事は `both` 推奨 |
| `draft` | 未完成は `true`。公開前に `false` |

## 4. 本文構成

### 推奨構成（作品別記事）

1. **リード** — 結論の要点＋「配信は変わるため公式で確認」を一言
2. **読み放題/聴き放題かの確認方法** — 手順を番号付きリスト
3. **作品の魅力・どんな人向けか**（任意だが推奨）
4. **対象外のときの代替** — 単品購入、他サブスク、関連作品
5. **まとめ** — 箇条書き3〜5点

### 推奨構成（比較・活用法）

1. リード（両者の違いを一言）
2. 料金・プラン比較（表を活用）
3. 向いている人（箇条書き）
4. 使い分け・併用
5. まとめ（表形式も可）

### 見出し

- H2: 大見出し（目次に載る）。`##` のみ使用（H1はページタイトルのみ）
- H3: 小見出し。深掘り用
- H2は1記事あたり3〜7個目安

### 文字数

| タイプ | 目安 |
|--------|------|
| 作品別 | 1,500〜3,000字 |
| 比較・活用法 | 2,500〜4,500字 |

薄い記事は統合・深掘りしてから公開する。

### 文体・表記

- **です・ます調**で統一
- 読者への呼びかけは「あなた」でも「〜したい方」でも可。一記事内で統一
- サービス名: **Kindle Unlimited** / **Audible** と表記する。**「KU」略称は使わない**（UI・本文・メタデータすべて）
- **絵文字は使わない**
- **グラデーション等のUI指示は記事に書かない**（サイト側の話）

## 5. E-E-A-T・信頼性

必ず含める要素:

- **一次情報の限界**: 「配信・料金は変更される。登録・購入前は Amazon / Audible 公式で確認」
- **体験・調査**: 可能なら「当編集部が確認した時点では〜」と書く。未確認の断定はしない
- **根拠**: 料金・プラン・確認手順は **[amazon-official-reference.md](./amazon-official-reference.md)** と公式ページ。競合ブログの数値だけを根拠にしない
- **更新**: 配信状況記事は `updatedDate` を付けてリライトする

禁止:

- 読み放題対象を**断定しすぎる**（「絶対読める」など）
- 公式と異なる料金・特典
- 競合他社の誹謗
- 医学・法律等の専門的断言

## 6. アフィリエイト・リンク

- **Kindle Unlimited** と **Audible** は、記事本文（`.prose`）内の**各サービス名の初出**に、ビルド時に自動で公式ページへのアソシエイトリンクが付く（`rehype-affiliate-first-mention`）
- 2回目以降の同一名称や、すでに手動でリンクした箇所には重ねて付かない
- 上記以外の Amazon リンクは手動で記述する場合、アソシエイト tag 付与を想定（ビルド時・手動URL生成時）
- 記事末尾の CTA・開示文はテンプレートが自動挿入されるため、**本文に重複して長文の開示を書かない**
- リンク文言は「公式ページで確認」など具体的に
- `rel="nofollow sponsored"` はコンポーネント側で付与済み

## 7. 内部SEO・回遊

- 関連する既存記事があれば、本文から **1〜3本** 自然にリンク（アンカーは説明的な文言）
- 同じ `tags` / `services` の記事が関連記事として自動表示されるよう `tags` を適切に付ける
- 画像: `ogImage` 未指定時はサイトデフォルト。専用OGが必要なら `public/` に置きパス指定

## 8. 品質チェックリスト（公開前）

- [ ] メインキーワードが title / description / 初段 / いずれかの H2 に含まれている
- [ ] 検索意図に答えている（結論が先に来ている）
- [ ] `services` と `cta` が内容と一致している
- [ ] 公式確認の注意書きがある
- [ ] 本文から関連記事へ **内部リンクが2本以上** ある（柱・比較・作品別は特に必須）
- [ ] **作品別**: 確認手順・対象外の代替・FAQ（またはよくある質問）がある
- [ ] **比較・柱記事**: 料金表に加え、勘違いの整理または選び方の早見表がある
- [ ] 誤字・ですますのぶれがない
- [ ] 絵文字がない
- [ ] `draft: false`、日付が正しい
- [ ] ファイル名（slug）が英小文字・ハイフンで意味が通る

採点基準・リライト時の改善パターンは [article-quality-rubric.md](./article-quality-rubric.md) を参照。

## 9. SERP競合分析（DuckDuckGo）

記事の骨子を決める前に、**検索上位の構成を分析**する。アマサブでは DuckDuckGo 経由で取得する CLI を用意している。

- まず `duck-duck-scrape` で検索（非公式）
- 失敗時は HTML 版 DuckDuckGo に自動フォールバック
- 各 URL の HTML から H2/H3 を抽出して構成を比較

※ DuckDuckGo の公式 Instant Answer API は SERP 一覧を返さないため、上記方式を使う。

```bash
npm run research -- "三体 Kindle Unlimited"
npm run research -- "Audible 使い方" --limit=8
```

### 出力

- `research/日時_クエリ.md` — 推奨 H2・網羅トピック・独自視点・各ページの見出し一覧
- `research/日時_クエリ.json` — 同上（機械可読）

### 執筆への使い方

1. レポートの **推奨記事構成（H2案）** をたたき台にする（競合の頻出見出しを反映）
2. **網羅すべきトピック** で抜け漏れがないか確認する
3. **独自視点の候補** から、アマサブならではの章を1〜2本入れる  
   （確認手順、配信変動の注意、Kindle Unlimited / Audible の使い分けなど）
4. レポートは自動取得のため、**必ず上位記事を1〜2本は人間が目視**する
5. 分析結果を満たしたうえで、本ガイドの文体・フロントマターに沿って執筆する

取得に失敗した URL はレポートに記載される。件数が少ない場合は時間をおいて再実行する。

## 10. 自動執筆・AI利用時の指示

AI・スクリプトで記事を生成する場合:

1. **本ファイル**と **[amazon-official-reference.md](./amazon-official-reference.md)** を読む
2. 新規記事なら **`npm run research -- "メインキーワード"`** を実行し、最新の `research/*.md` を読む
3. 上記チェックリストを出力前に自己検証する
4. 不確かな配信状況は「要確認」と明記し、確認手順を厚く書く
5. サンプル記事の丸写しは禁止。`src/content/posts/` の既存記事は文体参考のみ
6. 生成後は `npm run build` でビルドエラーがないか確認
